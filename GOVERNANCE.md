## Release Cadence

herald-rules follows a **quarterly release schedule** aligned with the
NCI EVS Controlled Terminology publication cycle:

| Quarter | Release Window | NCI EVS CT Alignment |
|---------|----------------|----------------------|
| Q1 | Last week of March | March CT release |
| Q2 | Last week of June | June CT release |
| Q3 | Last week of September | September CT release |
| Q4 | Last week of December | December CT release |

Releases are published within one week of the quarterly NCI EVS CT
publication to ensure herald always validates against the latest
controlled terminology.

## Version Numbering

Versions follow the format: **v{YYYY}.{Q}**

| Component | Meaning | Example |
|-----------|---------|---------|
| `YYYY` | Calendar year | 2026 |
| `Q` | Quarter (1-4) | 2 |

Examples:

- `v2026.1` -- Q1 2026 release (March)
- `v2026.2` -- Q2 2026 release (June)
- `v2026.3` -- Q3 2026 release (September)
- `v2026.4` -- Q4 2026 release (December)

### Hotfix versions

Hotfix releases between quarters use a patch suffix: **v{YYYY}.{Q}.{P}**

- `v2026.2.1` -- First hotfix after the Q2 2026 release.
- `v2026.2.2` -- Second hotfix.

Hotfixes are reserved for corrections to rules that produce incorrect
results (false positives or false negatives). New rules are not added in
hotfix releases.

## Release Process

### Pre-release (T-2 weeks)

1. **Feature freeze.** No new rules are merged after this point. Only
   bug fixes and documentation updates are accepted.

2. **CT refresh.** Run `Rscript inst/scripts/fetch-ct.R` to pull the
   latest NCI EVS Controlled Terminology.

3. **Source check.** Run `Rscript inst/scripts/check-updates.R` to
   verify whether any upstream sources have been updated since the
   last release. Document findings in the release notes.

### Release candidate (T-1 week)

4. **Build and validate.** Run the rebuild scripts, then validate all
   rules and test cases:

   ```bash
   Rscript inst/scripts/build-configs.R
   Rscript inst/scripts/build-manifest.R
   Rscript tests/validate-rules.R
   Rscript tests/validate-herald-rules.R
   Rscript tests/validate-define-rules.R
   ```

5. **Tag the release candidate:** `git tag v2026.2-rc1`

6. **Integration testing.** Install the release candidate in the herald
   package and run the full herald test suite:

   ```r
   devtools::test()
   devtools::check()
   ```

### Release (T-0)

7. **Tag the release:** `git tag v2026.2`

8. **Build artifacts for release:**

   ```bash
   Rscript inst/scripts/build-configs.R
   Rscript inst/scripts/build-master-csv.R
   Rscript inst/scripts/build-manifest.R
   ```

9. **Create a GitHub release** with the release notes.

10. **Update CHANGELOG.md** with the final release entry.

11. **Notify downstream.** Open a PR in the herald package to update the
    bundled rules version.

## Merge Policy

### Rule changes (engines/)

- **Two approvals required.** At least one reviewer must have domain
  expertise in the relevant standard.
- All CI checks must pass (YAML validation, test cases, no duplicates).
- The PR must include test cases for any new or modified executable rule.
- Merges use **squash and merge** to maintain a clean history.

### Configuration changes (configs/)

- **Two approvals required.**
- Changes must document which rules are added or removed from each profile.

### Documentation changes (*.md, inst/sources.json)

- **One approval required.**
- Merges use **squash and merge.**

### Script changes (inst/scripts/)

- **One approval required** from a maintainer.
- Scripts must be tested locally before merging.

### Controlled Terminology updates (ct/)

- **One approval required.**
- CT updates must be generated by the fetch script, not manually edited.
- The PR title must include the CT publication date.

## Decision-Making

### Consensus model

Decisions about rule interpretation, scope, and severity are made by
**lazy consensus**: a proposal is considered accepted if no objections
are raised within 5 business days of the proposal being posted.

### Escalation

If consensus cannot be reached:

1. The discussion is escalated to the maintainers.
2. Maintainers vote. A simple majority decides.
3. In case of a tie, the project lead has the deciding vote.

### Regulatory interpretation disputes

When contributors disagree about the correct interpretation of a
regulatory rule:

1. The source document (FDA TCG, SDTMIG, etc.) is the ultimate
   authority.
2. If the source document is ambiguous, the more conservative
   interpretation is adopted (i.e., the interpretation that is more
   likely to flag potential issues).
3. The rationale for the chosen interpretation must be documented in
   the rule's YAML file or in a linked issue.

## Roles and Responsibilities

### Contributors

Anyone who submits a pull request. Contributors must follow the
[CONTRIBUTING.md](CONTRIBUTING.md) guidelines.

### Reviewers

Individuals with domain expertise who review pull requests. Reviewers
are responsible for verifying regulatory accuracy, schema compliance,
and test coverage. Reviewers do not need commit access.

### Maintainers

Individuals with merge access. Maintainers are responsible for:

- Reviewing and merging pull requests
- Managing the release process
- Running the CT fetch and source check scripts
- Resolving escalated disputes
- Ensuring backward compatibility

### Project Lead

The project lead has final authority on architectural decisions,
release timing, and tie-breaking votes. The project lead is also
responsible for coordinating with the herald package maintainers.

## Hotfix Policy

Hotfix releases are issued between quarters only when:

1. A rule produces **false positives** that block legitimate submissions.
2. A rule produces **false negatives** that allow non-conformant data
   to pass without warning.
3. A critical **schema or infrastructure bug** prevents rules from
   loading.

Hotfixes follow an expedited review process:

- **One approval required** (from a maintainer).
- The fix must include a test case that demonstrates the bug and
  verifies the fix.
- The hotfix is cherry-picked to the release branch and tagged with
  the patch version (e.g., `v2026.2.1`).

## Upstream Synchronization

herald-rules tracks the following upstream sources:

| Source | Sync Frequency | Method |
|--------|----------------|--------|
| CDISC CORE | Each release | `inst/scripts/fetch-cdisc.R` (CDISC Library API) |
| NCI EVS CT | Quarterly | `inst/scripts/fetch-ct.R` |
| FDA TCG | As published | Manual review |
| FDA Business Rules | As published | Manual review |
| FDA Validator Rules | As published | Manual review |
| PMDA Validation | As published | Manual review |

Run `Rscript inst/scripts/check-updates.R` before each release to
identify upstream changes.

## Breaking Changes

A "breaking change" is any modification that changes the pass/fail
outcome of an existing rule for data that previously passed or failed.

Breaking changes are:

- **Allowed** in quarterly releases with documentation in CHANGELOG.md.
- **Not allowed** in hotfix releases (hotfixes may only fix incorrect
  behavior).
- **Must be announced** at least 2 weeks before the release in a
  GitHub discussion.
- **Must include migration guidance** when a rule ID changes or a rule
  is split/merged.

## Amendment

This governance document may be amended by a PR with approval from at
least two maintainers. Changes to the release cadence or version
numbering scheme require approval from the project lead.
